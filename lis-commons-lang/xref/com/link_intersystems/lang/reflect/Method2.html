105         if (memberIterator.hasNext()) {
106             // The first method returned will be this method, since we start at the declaring class.

231     /**
232      * @param method
233      * @return true, if the {@link Method} of this {@link Method2} overloads the
234      * given {@link Method2}. Overloading is determined as specified by
235      * the java specification section 8.4.9.
236      *
237      * <pre>
238      * <h2>8.4.9 Overloading</h2>
239      * If two methods of a class (whether both declared in the same class, or both inherited by a class, or one declared and one
240      * inherited) have the same name but signatures that are not override-equivalent, then the method name is said to be
241      * overloaded. This fact causes no difficulty and never of itself results in a compile-time error. There is no required
242      * relationship between the return types or between the throws clauses of two methods with the same name, unless their
243      * signatures are override-equivalent.
244      *
245      * Methods are overridden on a signature-by-signature basis.
246      *
247      * If, for example, a class declares two public methods with the same name, and a subclass overrides one of them, the subclass
248      * still inherits the other method.
249      *
250      * When a method is invoked (¬ß15.12), the number of actual arguments (and any explicit type arguments) and the compile-time
251      * types of the arguments are used, at compile time, to determine the signature of the method that will be invoked (¬ß15.12.2).
252      * If the method that is to be invoked is an instance method, the actual method to be invoked will be determined at run time,
253      * using dynamic method lookup (¬ß15.12.4).
254      * <hr/>
255      * <h2>8.4.10.2 Example: Overloading, Overriding, and Hiding</h2>
256      * In the example:
257      *
258      *     class Point {
259      *     	int x = 0, y = 0;
260      *     	void move(int dx, int dy) { x += dx; y += dy; }
261      *     	int color;
262      *     }
263      *     class RealPoint extends Point {
264      *     	float x = 0.0f, y = 0.0f;
265      *     	void move(int dx, int dy) { move((float)dx, (float)dy); }
266      *     	void move(float dx, float dy) { x += dx; y += dy; }
267      *     }
268      *
269      * the class RealPoint hides the declarations of the int instance variables x and y of class Point with its own float instance
270      * variables x and y, and overrides the method move of class Point with its own move method. It also overloads the name move
271      * with another method with a different signature (¬ß8.4.2).
272      *
273      * In this example, the members of the class RealPoint include the instance variable color inherited from the class Point, the
274      * float instance variables x and y declared in RealPoint, and the two move methods declared in RealPoint.
275      *
276      * Which of these overloaded move methods of class RealPoint will be chosen for any particular method invocation will be
277      * determined at compile time by the overloading resolution procedure described in ¬ß15.12.
278      * </pre>
279      * @since 1.0.0;
280      */
281     public boolean overloads(Method2 method) {
282         Method2 higher = method;
283         return isNameEqual(higher) && isReturnTypeAssignableFrom(higher)
284                 && !isInterfaceImplementation(higher)
285                 && !method.overrides(this);
286     }

288     /**
289      * @param method
290      * @return true, if the {@link Method} of this {@link Method2} overrides the
291      * given method. The override logic is based upon the java
292      * specification section 8.4.2 Method signature.
293      *
294      * <pre>
295      *         <h2>8.4.2 Method Signature</h2>
296      * It is a compile-time error to declare two methods with override-equivalent signatures (defined below) in a class.
297      *
298      * Two methods have the same signature if they have the same name and argument types.
299      *
300      * Two method or constructor declarations M and N have the same argument types if all of the following conditions hold:
301      *
302      * They have the same number of formal parameters (possibly zero)
303      * They have the same number of type parameters (possibly zero)
304      *
305      * Let <A1,...,An> be the formal type parameters of M and let <B1,...,Bn> be the formal type parameters of N. After renaming
306      * each occurrence of a Bi in N's type to Ai the bounds of corresponding type variables and the argument types of M and N
307      * are the same.
308      *
309      * The signature of a method m1 is a subsignature of the signature of a method m2 if either
310      *
311      * m2 has the same signature as m1, or
312      * the signature of m1 is the same as the erasure of the signature of m2.
313      *
314      * <hr/>
315      * <h3>Discussion</h3>
316      * The notion of subsignature defined here is designed to express a relationship between two methods whose signatures are
317      * not identical, but in which one may override the other.
318      *
319      * Specifically, it allows a method whose signature does not use generic types to override any generified version of that method.
320      * This is important so that library designers may freely generify methods independently of clients that define subclasses or
321      * subinterfaces of the library.
322      *
323      * Consider the example:
324      *
325      *    	class CollectionConverter {
326      *        	List toList(Collection c) {...}
327      *        }
328      *    	class Overrider extends CollectionConverter{
329      * List toList(Collection c) {...}
330      *    }
331      *
332      * Now, assume this code was written before the introduction of genericity, and now the author of class CollectionConverter decides
333      * to generify the code, thus:
334      *
335      *    	class CollectionConverter {
336      *    		<T> List<T> toList(Collection<T> c) {...}
337      *        }
338      *
339      * Without special dispensation, Overrider.toList() would no longer override CollectionConverter.toList(). Instead, the code
340      * would be illegal. This would significantly inhibit the use of genericity, since library writers would hesitate to migrate existing code.
341      *
342      * Two method signatures m1 and m2 are override-equivalent iff either m1 is a subsignature of m2 or m2 is a subsignature of m1.
343      *
344      * The example:
345      *
346      *    	class Point implements Move {
347      *    		int x, y;
348      *    		abstract void move(int dx, int dy);
349      *    		void move(int dx, int dy) { x += dx; y += dy; }
350      *        }
351      *
352      * causes a compile-time error because it declares two move methods with the same (and hence, override-equivalent) signature.
353      * This is an error even though one of the declarations is abstract.
354      * <hr/>
355      * <h2>8.4.10.1 Example: Overriding</h2>
356      * In the example:
357      *
358      *   	class Point {
359      *   		int x = 0, y = 0;
360      *   		void move(int dx, int dy) { x += dx; y += dy; }
361      *    }
362      *   	class SlowPoint extends Point {
363      *   		int xLimit, yLimit;
364      *   		void move(int dx, int dy) {
365      *   			super.move(limit(dx, xLimit), limit(dy, yLimit));
366      *        }
367      *   		static int limit(int d, int limit) {
368      *   			return d > limit ? limit : d < -limit ? -limit : d;
369      *        }
370      *    }
371      *
372      * the class SlowPoint overrides the declarations of method move of class Point with its own move method, which limits the
373      * distance that the point can move on each invocation of the method. When the move method is invoked for an instance of
374      * class SlowPoint, the overriding definition in class SlowPoint will always be called, even if the reference to the
375      * SlowPoint object is taken from a variable whose type is Point.
376      * </pre>
377      * @since 1.0.0;
378      */
379     public boolean overrides(Method method) {
380         Method2 referenceMethod = new Method2(method);
381 
382         if (referenceMethod.isDeclaringClassIsAnInterface()) {
383             return false;
384         }
385 
386         if (!isNameEqual(referenceMethod)) {
387             return false;
388         }
389 
390         if (!isAccessModifierOverriddingCompatible(referenceMethod)) {
391             return false;
392         }
393 
394         if (!isDeclaringClassAssignableFrom(referenceMethod)) {
395             return false;
396         }
397 
398         if (!areParametersEqual(referenceMethod)) {
399             return false;
400         }
401 
402         return isReturnTypeAssignableFrom(referenceMethod);
403     }
